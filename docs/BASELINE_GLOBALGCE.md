# GlobalGCE Baseline Protocol

## 1. GlobalGCE Source

GlobalGCE is reproduced from the official repository:

```text

```

The official source is kept under:

```text
baselines/globalgce_official
```

This mirrors the repository convention already used for:

```text
baselines/gcfexplainer_official
```

Do not use `external/GlobalGCE` for this project. Do not directly modify files inside `baselines/globalgce_official`; all wrappers, adapters, Slurm scripts, and reports belong to this repository's `scripts/`, `src/`, and `docs/` directories.

## 2. Why GlobalGCE Is A Suitable Baseline

GlobalGCE learns global graph counterfactual explanations through reusable graph transformation patterns. This makes it a suitable baseline for the current project because our final comparison is global counterfactual explanation coverage rather than only local single-graph recourse. GlobalGCE's output type is not identical to Ours: Ours is deletion-only selected subgraphs, while GlobalGCE can produce rule-like graph interventions and full counterfactual graph outputs.

Under the current project convention, GlobalGCE should therefore be evaluated with native-action CCRCov whenever possible, not forced into deletion-only evaluation.

## 5. Why Official Metrics Are Not Enough

GlobalGCE official validity, proximity, and comprehensibility are useful reproduction diagnostics, but they are not sufficient for the paper's horizontal comparison in this repository. The final comparison must be recomputed with:

- the same dataset split;
- the same frozen project teacher;
- the same K;
- the same distance threshold;
- the same cost function;
- the same native-action CCRCov protocol.

Official metrics should be recorded, but final reported CCRCov, CFDrop, FlipRate, Cost, SuppCov, StructRed, and CovRed must be generated by project-owned evaluation code.

## 7. Action-rule Adaptation

GlobalGCE outputs are adapted in two stages.

### native-cf-fullgraph

`native-cf-fullgraph` mode uses GlobalGCE's generated counterfactual graph
artifacts, exported from `cfs.pkl` files when available. Each CF graph is
converted through:

```text
GlobalGCE graph -> RDKit Mol -> canonical SMILES -> project teacher
```

The converted full counterfactual graphs are treated as baseline-generated
candidate molecules. The project teacher then recomputes:

- flip;
- CFDrop;
- cost;
- CCRCov@0.05;
- CCRCov@0.10;
- CCRCov@0.20.

This is the first-stage required unified evaluation mode. SuppCov is skipped in
this mode because complete CF graphs are not LHS/RHS support rules:

```text
suppcov_status = skipped_for_fullgraph_native_cf
```

When `distance_mode=tanimoto`, the output must record:

```text
distance_type = tanimoto_fingerprint
```

This is a smoke/diagnostic distance, not the final GREED-GED main line.

### native-cf-delta-action

`native-cf-delta-action` attempts to project a source graph and a CF graph into
an action difference. This requires reliable source-parent alignment and atom
mapping. If these mappings are missing, the evaluator must not fabricate an
action. It should report:

```text
projected_delta_supported = false
projected_delta_reason = missing_source_parent_mapping_or_atom_mapping
```

### rule-action

`rule-action` mode reads `globalgce_rules.jsonl`. It checks whether the LHS subgraph can match parent graphs and reports:

- SuppCov;
- StructRed;
- CovRed;
- per-rule match counts.

Safe RHS replacement requires attachment and boundary-edge handling. If this is
not available, the evaluator must report:

```text
rule_action_supported = false
unsupported_reason = unsafe_lhs_rhs_replacement_without_attachment_mapping
```

Do not hard-code a fake transformed molecule to make CCRCov non-null.

### Graph To SMILES To Teacher Audit

GlobalGCE official graph format is a GNN/frequent-subgraph graph format. Not
every exported graph is guaranteed to be chemically valid under RDKit. Reports
must include:

- valid graph-to-SMILES candidate rows;
- invalid candidate rows with reasons such as `unknown_node_label`,
  `unknown_edge_label`, `rdkit_add_bond_error`, `sanitize_error`,
  `valence_error`, `empty_graph`, or `malformed_record`;
- `valid_rate`;
- `cfs_smiles_ok_rate`;
- `teacher_eval_success_rate`.

## 12. Known Caveat: AIDS Label Alignment

GlobalGCE official preprocessing flips AIDS graph labels:

```text
y = 1 - y
```

It also stores internal node labels with a padding shift:

```text
internal_node_label = raw_aids_node_label + 1
0 = padding
```

The AIDS raw maps are:

```text
node label: 0 C, 1 O, 2 N, 3 Cl, 4 F, 5 S
edge label: 0 single, 1 double, 2 triple
class label: 0 a, 1 i
```

Therefore, do not assume GlobalGCE internal label=1 is identical to this
project's `HIV_active=1`. All GlobalGCE reports must preserve
`label_alignment_warning` and use the frozen project teacher for final CCRCov /
CFDrop / FlipRate evaluation. The current evaluation interpretation is:

```text
GlobalGCE official AIDS graph/export artifacts are baseline raw outputs.
Final project labels come from data/raw/AIDS/HIV.csv, column HIV_active.
```
